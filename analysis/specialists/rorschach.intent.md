## 1. Stylistic fingerprint

### Naming
- **Backend**: `snake_case` modules + functions, `PascalCase` classes, leading-underscore privates (`_reload_hat`, `_run_migrations`, `_apply_analysis`, `_get_setting`). Consistent both eras.
- **Routes**: `routes/<noun>.py`, services `services/<noun>_service.py`. Pre-v0.2 services all carry the `_service.py` suffix (case_service, hat_service, room_service, search_service); v0.2 services drop it (claude_analysis, hat_analysis_pipeline, background_removal, melin_recap, settings_service). Mixed convention — settings_service kept the suffix, the rest didn't. Mild inconsistency.
- **Frontend**: `PascalCase.tsx` page components, `camelCase` API helpers, `hr-` prefix on every custom CSS class (`.hr-hero`, `.hr-stat`, `.hr-carousel`, `.hr-analysis-status`). The `hr-` prefix is religious — every non-Bootstrap-mimicking class follows it. That's deliberate namespace hygiene.
- **CSS tokens**: All custom properties prefixed by domain — `--bg-*`, `--neon-*`, `--gradient-*`, `--font-*`, `--shadow-*`, `--glow-*`. Section dividers in `tokens.css` use `/* === Header ====================== */` ASCII boxes. This is someone who likes their stylesheets to feel like a typography spec sheet.

### Comment density and tone
- **Pre-v0.2.0**: minimal. case_service.py has one line of comment ("Unassign all hats before deleting", line 86). hat_service.py has inline comments only on the column type fields (`# new_with_tags, new, worn`, `# small, classic, x_large`). No module docstrings.
- **v0.2.0**: every new service module starts with a triple-quoted docstring that explains *why*, not *what*. Examples:
  - `background_removal.py:1-6` — explains the u2netp choice and how to swap models
  - `melin_recap.py:1-9` — explains *why* there's no scraping (JS-rendered, brittle on a Pi) and *why* prices stay null (no fabrication)
  - `claude_analysis.py:1-7` — explains the tool-use shape and prompt caching
  - `database.py:17-18` — explains why the migration DDL strings are hard-coded literals (semgrep finding)
- **Tone shift**: pre-v0.2 commits are casual ("Spruced up the web UI", "Made updates to allow you to edit colors after identification"). v0.2 commits are conventional-commits format with structured bodies (`feat(branding):`, `fix(docker):`, `fix(ui):`) and explain trade-offs in the body.

### File length norms
- Backend services pre-v0.2: 50-180 lines, single responsibility, no stratification beyond service-per-noun.
- v0.2 services: similarly compact (claude_analysis 263 lines is the outlier, justified by the tool schema literal). The author resists letting any single file balloon.
- Frontend pages: 100-300 lines. HatDetailPage (304 lines) is the heaviest because it has nested status/price components defined inline. EditHatPage (298 lines) similar. CSS is the only thing that breaks the discipline (1278 lines in app.css, but it's deliberately one file because dropping Bootstrap meant rebuilding everything).

### Error handling style
- Pre-v0.2: services raise `HTTPException` directly. The HTTP layer leaks into the service layer (e.g. `case_service.py:65`). It's pragmatic and fine for a single-developer FastAPI app, but it's not "clean architecture."
- v0.2: domain exceptions (`ClaudeAnalysisError`) translated at the boundary, with explicit `from exc` chaining for traceability (claude_analysis.py:209-214). Two `except Exception` cases are both annotated `# noqa: BLE001 — surfaced to UI` with a one-line justification.
- **Inconsistency**: the v0.2 settings/Claude code uses domain exceptions, but the photo routes still raise `HTTPException(400)` directly (routes/hats.py:149, 184-186). The new pattern was applied to new code but old patterns were left alone — sensible, not principled.

## 2. Idiomatic alignment

The code generally **moves with** its frameworks, not against them.

### FastAPI: idiomatic
- Routers split by noun (`routes/cases.py`, `routes/hats.py` …), composed in `routes/__init__.py`. No god-router.
- `Depends(get_db)` everywhere; no global session.
- `response_model=` on every route → automatic schema generation, OpenAPI docs come for free.
- Lifespan context manager for startup (`app.py:43-52`). The right modern pattern.
- SPA fallback at `app.py:83-89` — clean. Mounts `/uploads` and `/assets` separately so they don't collide with the catch-all.

### SQLAlchemy 2.x async: idiomatic-with-known-gotcha-handling
- `Mapped[T]` typing, `mapped_column()`, `DeclarativeBase` — all the 2.x style.
- `selectinload()` everywhere relationships cross async boundaries — correct.
- The `_reload_X` pattern (`db.expire_all()` then re-query with selectinload) is *fighting* SQLAlchemy a bit but for a real reason: after a commit + relationship mutation in async mode, lazy access blows up. The author learned this the hard way and codified the workaround into a private helper per service. The CLAUDE.md and MEMORY.md both call this out explicitly — it's a known scar tissue.
- One sharp edge handled well: `room_service.delete_room` uses `update(Case).where(...)` instead of touching ORM relationships before delete (room_service.py:64-67), with a comment explaining "to avoid cascade issues." That's experience talking.

### React / TanStack Query: idiomatic
- `useQuery`/`useMutation` everywhere, `queryClient.invalidateQueries({ queryKey: [...] })` after mutations, no manual cache writes.
- Keys are tuples (`['hat', id]`, `['settings', 'logo']`, `['meta', 'styles']`) — proper hierarchical cache.
- The CLAUDE.md explicitly documents the `['rooms']` vs `['meta', 'rooms']` key distinction and warns to invalidate both — the author has been bitten by this and wrote it down.
- No prop drilling worth mentioning; query state is the primary state. `useState` is reserved for UI-local concerns (drafts, modal toggles, carousel index).

### TypeScript: pragmatic, not paranoid
- Types in one shared `types/index.ts` mirror Pydantic — sensible for a small app.
- API helpers use `Record<string, unknown>` for create payloads (api/hats.ts:13) instead of typed DTOs. That's a corner cut — it works but TS won't catch a typo'd field name. Acceptable for a one-person project.
- `apiFetch<T>` (api/client.ts) is a 16-line generic wrapper. No axios, no react-query mutation factory. Right-sized.

### CSS: opinionated, not framework-driven
- The token system (`tokens.css`) is the cleanest file in the repo. Variables are grouped by domain with ASCII section headers. This is someone who knows design tokens.
- `app.css` reimplements ~100 lines of Bootstrap utilities (lines 1176-1278) so the JSX can keep using Bootstrap class names. Pragmatic compatibility shim — avoids touching every component file when dropping the dependency.
- iOS-specific touches (`font-size: 16px` on inputs to prevent zoom; `env(safe-area-inset-bottom, 0px)`; `viewport-fit=cover` mentioned in changelog) — someone tested this on actual hardware.

### Where the code fights its frameworks
- `claude_analysis.py:18` imports `from anthropic._exceptions import AuthenticationError` — reaches into the SDK's private namespace. The public surface re-exports it. Fragile.
- `routes/hats.py:188-194` does function-local imports inside `reanalyze_hat`. There's no circular-import reason; the imports are just defensive.
- `frontend/src/api/hats.ts:13` types create payloads as `Record<string, unknown>` — chooses to lose type safety rather than maintain a parallel set of "create input" types.

  3. `hat_analysis_pipeline.py:66-70` makes `analysis_status='skipped'` a first-class state distinct from `error`.

- **Reanalyze route**. routes/hats.py:177-213 inlines all the orchestration (function-local imports, manual error path, manual commit) instead of calling a shared helper alongside `finalize_hat_photo`. ## 5. Tacit assumptions

The code assumes, without ever stating:

- **Single user.** No `User` model, no auth, no per-user data. The `AppSetting` table is global. If two people share a Pi they share the API key, the logo, all the data.
- **Trusted network.** No CSRF, CORS allows `localhost:5173` for dev, no rate limiting on the upload route or the `/api-key/test` endpoint (which calls Claude on every press, costing the user money).
- **No concurrent uploads.** The rembg session is module-global with a single `asyncio.Lock` (background_removal.py:21, 55). Two simultaneous uploads serialise. Fine for one user, would be a bottleneck otherwise.
- **No concurrent writes to `app_settings`.** `_set_setting` (settings_service.py:24-35) does check-then-update without a transaction guard; two writers race. SQLite + single user makes this a non-issue.
- **Image data is well-formed.** `Image.open()` is called without try/except in `utils/photo.py:22`. A malformed image will 500. The route does check the content type but not the magic bytes.
- **Claude returns valid JSON in the tool block.** `claude_analysis.py:222-242` parses defensively (`KeyError, TypeError, ValueError`) but assumes the tool was actually called. If Claude returns text instead of tool_use the code raises "Claude did not return a tool_use block" — handled — but the prompt relies on `tool_choice={"type": "tool", "name": "..."}` doing the right thing.
- **Filesystem is writable and persistent.** The lifespan creates `uploads/cases`, `uploads/hats`, `uploads/branding` (app.py:46-49). Failure to create those just crashes startup.
- **The default room (id=1) always exists.** Cases default `room_id=1` (case.py:17). `delete_room` refuses to delete it (room_service.py:58-61). But there's no DB constraint enforcing the row exists — `ensure_default_room()` (database.py:93-101) re-creates it on startup. Survivable but implicit.
- **Hat photos are smaller than 1200px after Pillow's thumbnail.** `MAX_DIMENSION = 1200` in utils/photo.py:6 is hard-coded; the Claude vision call sends the entire base64-encoded image regardless of size. No max-bytes check before sending.
- **The Anthropic SDK's exception classes are stable.** `from anthropic._exceptions import AuthenticationError` (claude_analysis.py:18) reaches into a private namespace.
